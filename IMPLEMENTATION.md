# Деталі реалізації Smart Climate Controller

## Де знаходиться кожна ключова логіка

### 1. Синхронізація термостатів у групі
**Файл:** `coordinator.py`
**Методи:**
- `_synchronize_group_modes` (рядки 508-538)
- `set_room_hvac_mode` (рядки 486-506)

**Як працює:**
Коли користувач змінює режим одного термостату:
1. `climate.py` викликає `coordinator.set_room_hvac_mode(room_name, new_mode)`
2. Coordinator оновлює стан кімнати
3. Якщо кімната входить у групу, викликається `_synchronize_group_modes`
4. Метод знаходить всі інші кімнати в цій групі
5. Для кожної кімнати (крім тієї що OFF) встановлюється той самий режим
6. Зберігається timestamp зміни режиму

**Важливо:** Синхронізуються саме режими термостатів (heat/cool/auto), а не фізичні режими кондиціонерів.

### 2. Вибір фактичного режиму кондиціонерів
**Файл:** `coordinator.py`
**Методи:**
- `_determine_physical_mode` (рядки 216-261)
- `_determine_group_hvac_mode` (рядки 189-214)

**Як працює:**
1. Спочатку визначається групувальний режим термостатів (`_determine_group_hvac_mode`):
   - Шукає останній змінений режим серед активних термостатів
   - Якщо всі OFF → повертає OFF
   - Інакше → повертає найсвіжіший режим (heat/cool/auto)

2. Потім визначається фізичний режим (`_determine_physical_mode`):
   - Якщо груповий режим HEAT → фізичний HEAT
   - Якщо груповий режим COOL → фізичний COOL
   - Якщо груповий режим AUTO:
     - Читається температура на вулиці
     - Якщо < heat_only_threshold → HEAT
     - Якщо > cool_only_threshold → COOL
     - Інакше → викликається `_determine_mode_in_transition_zone`

### 3. Алгоритм вибору режиму в перехідній зоні
**Файл:** `coordinator.py`
**Метод:** `_determine_mode_in_transition_zone` (рядки 263-345)

**Як працює:**
1. **Перевірка інтервалу:**
   - Шукає найсвіжішу зміну режиму серед кімнат групи
   - Якщо з моменту зміни пройшло < `min_mode_switch_interval` секунд → зберігає поточний режим

2. **Аналіз потреб кімнат (scoring):**
   - Для кожної активної кімнати обчислюється різниця між поточною та цільовою температурою
   - Якщо `temp_diff < -major_deviation_threshold` → heat_need_score += 10 (критична потреба в обігріві)
   - Якщо `temp_diff < 0` → heat_need_score += 5 (помірна потреба в обігріві)
   - Якщо `temp_diff > major_deviation_threshold` → cool_need_score += 10 (критична потреба в охолодженні)
   - Якщо `temp_diff > 0` → cool_need_score += 5 (помірна потреба в охолодженні)

3. **Прийняття рішення:**
   - Якщо heat_need_score > cool_need_score → HEAT
   - Якщо cool_need_score > heat_need_score → COOL
   - Якщо рівні і є поточний режим → зберігає поточний
   - Якщо рівні і немає поточного → COOL (default)

**Це гарантує:**
- Режим не змінюється занадто часто
- Враховуються потреби всіх кімнат
- Критичні відхилення мають пріоритет
- При рівних потребах зберігається стабільність

### 4. Логіка корекції температури (дрібна та сильна)
**Файл:** `coordinator.py`
**Метод:** `_control_room_temperature` (рядки 347-422)

**Параметри:**
- `minor_correction_hysteresis` - поріг для дрібної корекції (default 0.5°C)
- `minor_correction_value` - значення дрібної корекції (default 5°C)
- `major_deviation_threshold` - поріг для сильної корекції (default 1°C)
- `major_correction_value` - значення сильної корекції (default 10°C)

**Логіка для HEAT режиму:**
```
temp_diff = indoor_temp - target_temp

if temp_diff > major_deviation_threshold:
    # Перегрів - вимкнути
    turn_off_ac()
elif temp_diff < -major_deviation_threshold:
    # Сильний холод - сильна корекція
    set_ac_temp(target + major_correction)  # target + 10°C
elif temp_diff < -minor_hysteresis:
    # Легкий холод - дрібна корекція
    set_ac_temp(target + minor_correction)  # target + 5°C
else:
    # Норма
    set_ac_temp(target)
```

**Логіка для COOL режиму:**
```
temp_diff = indoor_temp - target_temp

if temp_diff < -major_deviation_threshold:
    # Переохолодження - вимкнути
    turn_off_ac()
elif temp_diff > major_deviation_threshold:
    # Сильна спека - сильна корекція
    set_ac_temp(target - major_correction)  # target - 10°C
elif temp_diff > minor_hysteresis:
    # Легка спека - дрібна корекція
    set_ac_temp(target - minor_correction)  # target - 5°C
else:
    # Норма
    set_ac_temp(target)
```

**Чому це працює:**
- Кондиціонер не просто вмикається/вимикається, а працює постійно з коректованою температурою
- Чим більше відхилення, тим агресивніше корекція
- При перегріві/переохолодженні кондиціонер вимикається (економія енергії)
- Гістерезис запобігає постійним змінам при коливаннях

### 5. Обмеження частоти перемикань
**Файл:** `coordinator.py`
**Метод:** `_control_room_climate` (рядки 424-519)

**Параметри:**
- `min_mode_switch_interval` - мінімальний інтервал між зміною режимів heat/cool (default 600 сек = 10 хв)
- `min_power_switch_interval` - мінімальний інтервал між вмиканням/вимиканням (default 300 сек = 5 хв)

**Як працює:**
1. **Визначення змін:**
   - Читає поточний стан кондиціонера
   - Порівнює з бажаним станом
   - Визначає чи потрібна зміна режиму або температури

2. **Перевірка інтервалу зміни режиму:**
   ```python
   if mode_changed:
       time_since_switch = (now - room_state.last_mode_switch).total_seconds()
       if time_since_switch < min_mode_interval:
           # Пропустити команду
           return
   ```

3. **Перевірка інтервалу вмикання/вимикання:**
   ```python
   if power_changed:
       time_since_switch = (now - room_state.last_power_switch).total_seconds()
       if time_since_switch < min_power_interval:
           # Пропустити команду
           return
   ```

4. **Виконання команди:**
   - Викликає `climate.set_hvac_mode` або `climate.set_temperature`
   - Оновлює timestamps в `RoomState`

**Результат:**
- Кондиціонер не перемикається занадто часто
- Захист компресора від частих вмикань/вимикань
- Стабільна робота без "смикання"

### 6. Відновлення стану після перезапуску
**Файл:** `coordinator.py`
**Методи:**
- `_save_state` (рядки 565-584)
- `_restore_state` (рядки 586-612)

**Що зберігається:**
```python
{
    "rooms": {
        "Living Room": {
            "target_temperature": 22.0,
            "hvac_mode": "heat",
            "last_mode_switch": "2026-04-16T10:30:00+00:00",
            "last_power_switch": "2026-04-16T10:25:00+00:00"
        },
        ...
    }
}
```

**Коли зберігається:**
- Після кожного update циклу (кожні 30 секунд)
- При shutdown інтеграції

**Коли відновлюється:**
- При першому запуску інтеграції (`async_config_entry_first_refresh`)

**Технологія:**
- Використовується `Store` з Home Assistant
- JSON файл в `.storage/smart_climate_controller.storage`
- Версія 1 (для майбутньої міграції)

**Що дає:**
- Після перезапуску HA термостати зберігають режим та цільову температуру
- Інтервали перемикань не скидаються (запобігає зайвим командам одразу після старту)
- Користувачу не потрібно повторно налаштовувати

## Порядок виконання в update циклі

Кожні 30 секунд coordinator викликає `_async_update_data`:

1. **Отримання груп** (`_get_multi_split_groups`):
   - Розбиває кімнати на групи за `multi_split_group`
   - Кімнати без групи отримують унікальну групу

2. **Обробка кожної групи** (`_process_group`):

   a) Визначення групового режиму термостатів (`_determine_group_hvac_mode`)

   b) Якщо OFF → вимкнути всі кондиціонери

   c) Визначення фізичного режиму (`_determine_physical_mode`)
      - Враховує зовнішню температуру
      - Викликає `_determine_mode_in_transition_zone` якщо AUTO і в перехідній зоні

   d) Для кожної кімнати в групі (`_control_room_temperature`):
      - Аналіз відхилення температури
      - Вибір типу корекції (minor/major/off)
      - Виклик `_control_room_climate`

   e) Виконання команд (`_control_room_climate`):
      - Перевірка інтервалів
      - Відправка команд до кондиціонера
      - Оновлення timestamps

3. **Збереження стану** (`_save_state`)

## Приклад сценарію

**Умови:**
- Група "main" з 2 кімнат: Спальня та Вітальня
- Обидві в режимі AUTO
- Цільова температура: 22°C
- Зовнішня температура: 18°C (перехідна зона між 10°C та 23°C)
- Поточна температура Спальня: 20°C
- Поточна температура Вітальня: 23°C

**Що станеться:**

1. **Групувальний режим:** AUTO (обидві кімнати в AUTO)

2. **Вибір фізичного режиму:**
   - Температура 18°C в перехідній зоні
   - Викликається `_determine_mode_in_transition_zone`
   - Спальня: temp_diff = 20 - 22 = -2°C → heat_need_score += 10
   - Вітальня: temp_diff = 23 - 22 = +1°C → cool_need_score += 10
   - Рівні скори (10:10) → зберігається поточний режим
   - Якщо немає поточного → default COOL

3. **Припустимо вибрано HEAT:**
   - Спальня:
     - temp_diff = -2°C < -major_threshold (-1°C)
     - Сильна корекція: target + 10°C = 32°C
     - Кондиціонер переключається в HEAT, 32°C
   - Вітальня:
     - temp_diff = +1°C = major_threshold
     - Перегрів в режимі HEAT → вимкнути
     - Кондиціонер вимикається

4. **Через 5 хвилин:**
   - Спальня прогрівається до 21.5°C
   - temp_diff = -0.5°C (в межах hysteresis)
   - Дрібна корекція: target + 5°C = 27°C
   - Температура на кондиціонері знижується до 27°C

5. **Через 10 хвилин:**
   - Спальня прогріта до 22°C
   - temp_diff = 0°C
   - Норма: target = 22°C
   - Кондиціонер тримає 22°C

**Результат:** Спальня комфортно прогріта, Вітальня не перегрівається, система стабільна.

## Можливі сценарії розширення

### 1. Додавання сенсорів
Можна додати sensor платформу в `__init__.py`:
```python
PLATFORMS = [Platform.CLIMATE, Platform.SENSOR]
```

Створити `sensor.py` з сенсорами:
- Поточний фізичний режим групи
- Час до наступного можливого перемикання
- Скор потреби в heat/cool

### 2. Додавання сервісів
Створити `services.yaml` з сервісами:
- Форсувати перемикання режиму (ігнорувати інтервали)
- Синхронізувати групу вручну
- Скинути стан

### 3. Підтримка інших пристроїв
Замість прив'язки до climate домену, використовувати адаптери:
```python
class DeviceAdapter:
    def set_mode(self, mode): ...
    def set_temperature(self, temp): ...
    def get_state(self): ...
```

### 4. Складніші алгоритми
- Машинне навчання для передбачення потреб
- Інтеграція з прогнозом погоди
- Графіки споживання енергії
- Оптимізація за вартістю електроенергії
